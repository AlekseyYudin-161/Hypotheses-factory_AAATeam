# Норникель AI Science Hack 2026
**Задача: Фабрика гипотез** 

Инструмент на базе ИИ, который по целевому KPI и базе знаний генерирует, ранжирует и обосновывает проверяемые научно-исследовательские гипотезы для обогащения руд и металлургического передела. 

Вход принимает: 1) описание технологической проблемы / KPI; 2) технологические ограничения; 3) опциональные документы (таблицы, отчёты, схемы). 

Выход — ранжированный список гипотез с механизмом, оценками новизны/риска/ценности, экономической оценкой, причинно-следственной цепочкой и ссылками на источники. Решение интерпретируемо и допускает экспертную корректировку (human-in-the-loop).

## Архитектура

Система состоит из трёх слоёв:

- **Данные и индексация** (офлайн-подготовка корпуса + приём файлов пользователя)
- **RAG-ядро** (генерация, поиск, ранжирование) — [rag/README.md](rag/README.md).
- **Интерфейс** (Streamlit-верстак, экспертная правка, экспорт).

Всё разворачивается одним `docker compose` (приложение + PostgreSQL/pgvector).


## Данные и индексация (data engineering)

Отвечает за то, чтобы разнородные исходные данные превратились в базу знаний, пригодную для поиска, и за приём данных, загружаемых вместе с запросом.

- **Хранилище** — PostgreSQL + pgvector: чанки корпуса с 1536-мерными эмбеддингами, полнотекстовый индекс для гибридного поиска, триплеты «материал–условие–эффект» и метаданные источников для провенанса.
- **Офлайн-индексация корпуса** — парсинг таблиц (xlsx/csv) и отчётов, разбиение на чанки, извлечение эмбеддингов, запись в pgvector.
- **Приём файлов с запросом** — загруженные пользователем таблицы, PDF и изображения обрабатываются на лету: таблицы и текст извлекаются напрямую, а с технологических схем и чертежей факты снимаются мультимодальной моделью (vision). Данные запроса не смешиваются с корпусом.
- **Абстракция LLM-провайдера** — доступ к генерации и эмбеддингам изолирован за единым интерфейсом, что позволяет за один правочный цикл сменить провайдера в ходе работы без переписывания пайплайна.
- **Интерпретация предметных данных** (минералогия хвостов, классы крупности, соответствие элементов и минералов, экономика по ценам металлов) выполнена при консультациях со специалистом-хемоинформатиком @Tupanvirus.

## Интерфейс (Streamlit)

Верстак, в котором пользователь работает с системой: ввод KPI и ограничений, загрузка документов, выбор базы знаний. Результат отображается ранжированным списком гипотез — каждая с механизмом, оценками по трём осям, экономической оценкой, причинной цепочкой и ссылками на источники. Эксперт прямо в интерфейсе принимает/отклоняет гипотезы, отмечает проверенные, ставит оценку и комментарий (human-in-the-loop), а также экспортирует результат в DOCX, CSV, JSON и CSV для Jira/YouTrack. Интерфейс на русском, английском и китайском.

## Возможности

- Ввод KPI и технологических ограничений текстом; загрузка дополнительных документов (PDF, DOCX, TXT, MD, CSV, XLSX, JSON).
- Выбор базы знаний: научные публикации, патенты, внутренние отчёты, исторические эксперименты, данные о материалах и процессах, открытые источники.
- Для каждой гипотезы: механизм, новизна, риск, технологическая и экономическая ценность, причинно-следственная цепочка, происхождение с ссылками на источники, рекомендация по проверке.
- Ранжирование по трём осям с изменяемыми весами; корректный ответ «улучшать нечего», если он обоснован.
- Экспертная правка: принять/отклонить, отметить проверенной, поставить оценку и комментарий (сохраняется в БД).
- Экспорт: DOCX, CSV, JSON, CSV для Jira/YouTrack.
- Интерфейс на русском, английском и китайском.

## Запуск

**[Live demo (Streamlit Cloud)](https://hypothesis-workbench-aaa-team.streamlit.app)** - Публичное демо интерфейса и генерации гипотез. Работает на документах, загружаемых пользователем, и открытых источниках. Полный корпусный RAG (PostgreSQL/pgvector) в облачной версии `не подключён`.

**Полная версия (Docker):** подключает корпусную базу знаний — провенанс гипотез ссылается на источники корпуса.

```bash
docker compose up --build
```
Приложение откроется на `http://localhost:8501`.


**Локально (Python 3.12):**
```bash
git clone https://github.com/AlekseyYudin-161/Hypotheses-factory_AAATeam.git

cd Hypotheses-factory_AAATeam

cp .env.example .env

nano .env                                       # Заполните .env файл

python3 -m venv .venv

source .venv/bin/activate                       # On Windows use: .venv\Scripts\activate

python -m pip install -r requirements.txt

streamlit run app.py
```

## Технический контракт backend

Фронтенд обращается к ядру по адресу из переменной окружения:
```text
HYPOTHESIS_API_URL=http://localhost:8000/generate
```

Запрос:
```json
{
  "kpi": "Повысить извлечение никеля из руды на 8%",
  "constraints": "Не увеличивать расход реагентов и использовать существующее оборудование",
  "language": "ru",
  "knowledge_bases": ["Все базы знаний"],
  "use_open_sources": true,
  "industrial_scale": true
}
```

Backend сам определяет количество релевантных гипотез. Каждая гипотеза содержит поля механизма, трёх осей оценки, экономической оценки, причинной цепочки, оценки ресурсов и массива `evidence` с обязательным `source_url` для каждого источника. Полная схема ответа — в [rag/README.md](rag/README.md).

### Краткая суть модуля RAG

Ядро генерирует гипотезы по доизвлечению цветных металлов и снижению затрат для обогатительно-металлургического передела. На вход подаётся «богатый промпт»: текстовое описание проблемы, бюджет и сроки, Markdown-таблица материального баланса и опционально фото или схемы установки. На выходе — топ-3 отранжированные гипотезы, каждая с причинно-следственной цепочкой в формате PMDco, ссылками на источники по DOI и численными оценками по трём осям: Risk, Value, Novelty. Если улучшать нечего, ядро возвращает легальный отказ — это часть контракта, а не ошибка.

Идея RAG состоит в том, чтобы не просить LLM придумывать решение из головы, а извлекать из базы знаний реальные успешные эксперименты и заставлять модель переносить их механизмы по аналогии на целевой материал, оставаясь строго в рамках бюджета и цитируя источники.

Вся работа выполняется пайплайном из четырёх шагов под жёстким дедлайном 120 секунд:

1. **Декомпозиция входа** — LLM раскладывает «богатый промпт» на промышленные ограничения: что перерабатываем, что улучшаем, бюджет, сроки, масса и список механизмов.
2. **Гибридный поиск контекста** — по каждому механизму выполняется поиск в pgvector, сочетающий векторное и полнотекстовое ранжирование с объединением через Reciprocal Rank Fusion и дедупликацией по DOI.
3. **Синтез гипотез** — один вызов LLM по найденному контексту возвращает ровно три гипотезы с цепочкой, ключевым процессом, источниками и ожидаемым приростом KPI.
4. **Ранжирование и отсев** — по формуле `0.30·Novelty + 0.40·(1−Risk) + 0.30·Value`, с отсевом ниже порога и срезом до топ-3.

База знаний построена на Postgres с расширением pgvector: документы, чанки корпуса с 1536-мерными эмбеддингами и граф триплетов для оценки новизны. Компоненты внедряются через конструктор, поэтому LLM, клиент базы и модуль экономиста можно подменять на офлайн-заглушки для прогонов без БД и ключей. Адаптер [service.py](rag/service.py) конвертирует результат ядра в «богатую» гипотезу под формат фронтенда.

## Хранение

Локально экспертные оценки и гипотезы сохраняются в `data/workbench.db` (SQLite). Для постоянного хранения между перезапусками контейнера подключается Postgres или постоянный volume.
