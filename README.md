# Верстак гипотез

Streamlit-интерфейс для генерации, ранжирования, экспертной проверки и экспорта научно-исследовательских гипотез.

## Реализовано

- KPI и технологические ограничения вводятся текстом;
- количество гипотез определяется генератором автоматически;
- механизм, новизна, риск, технологическая и экономическая ценность;
- ранжирование с изменяемыми весами трёх осей;
- учёт ограничений и промышленного масштаба;
- причинная цепочка и диаграмма влияния;
- оценка неопределённости и рекомендация по проверке;
- происхождение гипотезы с обязательными ссылками на источники;
- открытые источники разрешены и передаются backend через `use_open_sources=true`;
- эксперт может принять/отклонить гипотезу, отметить её проверенной, поставить оценку и оставить комментарий;
- текст гипотезы и системные оценки эксперт не редактирует;
- экспертные оценки сохраняются в SQLite;
- экспорт PDF, DOCX, CSV, JSON и CSV для Jira/YouTrack;
- русский, английский и китайский интерфейс;
- Docker и Docker Compose.

## RAG-ядро (backend)

Полное описание — в [rag/README.md](rag/README.md). Кратко о том, что делает модуль.

Ядро генерирует гипотезы по доизвлечению цветных металлов и снижению затрат для обогатительно-металлургического передела. На вход подаётся «богатый промпт»: текстовое описание проблемы, бюджет и сроки, Markdown-таблица материального баланса и опционально фото или схемы установки. На выходе — топ-3 отранжированные гипотезы, каждая с причинно-следственной цепочкой в формате PMDco, ссылками на источники по DOI и численными оценками по трём осям: Risk, Value, Novelty. Если улучшать нечего, ядро возвращает легальный отказ — это часть контракта, а не ошибка.

Идея RAG состоит в том, чтобы не просить LLM придумывать решение из головы, а извлекать из базы знаний реальные успешные эксперименты и заставлять модель переносить их механизмы по аналогии на целевой материал, оставаясь строго в рамках бюджета и цитируя источники.

Вся работа выполняется пайплайном из четырёх шагов под жёстким дедлайном 120 секунд:

1. **Декомпозиция входа** — LLM раскладывает «богатый промпт» на промышленные ограничения: что перерабатываем, что улучшаем, бюджет, сроки, масса и список механизмов.
2. **Гибридный поиск контекста** — по каждому механизму выполняется поиск в pgvector, сочетающий векторное и полнотекстовое ранжирование с объединением через Reciprocal Rank Fusion и дедупликацией по DOI.
3. **Синтез гипотез** — один вызов LLM по найденному контексту возвращает ровно три гипотезы с цепочкой, ключевым процессом, источниками и ожидаемым приростом KPI.
4. **Ранжирование и отсев** — по формуле `0.30·Novelty + 0.40·(1−Risk) + 0.30·Value`, с отсевом ниже порога и срезом до топ-3.

База знаний построена на Postgres с расширением pgvector: документы, чанки корпуса с 1536-мерными эмбеддингами и граф триплетов для оценки новизны. Компоненты внедряются через конструктор, поэтому LLM, клиент базы и модуль экономиста можно подменять на офлайн-заглушки для прогонов без БД и ключей. Адаптер [service.py](rag/service.py) конвертирует результат ядра в «богатую» гипотезу под формат фронтенда.

## Локальный запуск

```bash
py -3.14 -m pip install -r requirements.txt
py -3.14 -m streamlit run app.py
```

Для Python 3.12:

```bash
python -m pip install -r requirements.txt
streamlit run app.py
```

## Подключение backend

Задайте переменную окружения:

```text
HYPOTHESIS_API_URL=http://localhost:8000/generate
```

Запрос:

```json
{
  "kpi": "Повысить извлечение никеля из руды на 8%",
  "constraints": "Не увеличивать расход реагентов и использовать существующее оборудование",
  "language": "ru",
  "use_open_sources": true,
  "industrial_scale": true
}
```

Backend самостоятельно определяет количество релевантных гипотез.

Каждая гипотеза должна содержать:

```json
{
  "hypothesis": "...",
  "rationale": "...",
  "mechanism": "...",
  "expected_effect": "...",
  "industrial_scale": "...",
  "kpi_link": "...",
  "constraints_fit": "...",
  "novelty": {"score": 0.8, "why": "..."},
  "risk": {"score": 0.3, "why": "..."},
  "value": {"score": 0.9, "why": "..."},
  "economic_value": {"score": 0.8, "why": "..."},
  "success_probability": {"score": 0.3, "why": "..."},
  "verification_recommendation": "...",
  "causal_chain": ["воздействие", "механизм", "промежуточный эффект", "KPI"],
  "resource_estimate": {"time": "...", "cost": "...", "volume": "..."},
  "evidence": [
    {
      "triplet": "...",
      "source": "...",
      "source_url": "https://...",
      "page": 125,
      "chunk_id": "CH-001",
      "evidence_fragment": "...",
      "confidence": 0.87
    }
  ],
  "roadmap": [],
  "status": "draft"
}
```

`source_url` обязателен для каждого источника.

## Docker

```bash
docker compose up --build
```

Приложение откроется на `http://localhost:8501`.

## Хранение

Локально данные сохраняются в `data/workbench.db`. Для постоянного облачного хранения после перезапусков контейнера следует подключить Postgres или постоянный volume.


## Выбор базы знаний

В выпадающем поле можно выбрать:

- «Все базы знаний»;
- одну базу;
- несколько баз.

Поддерживаются научные публикации, патенты, внутренние отчёты, исторические эксперименты, данные о материалах, данные о процессах и открытые источники. В API передаётся массив `knowledge_bases`.

Показатель `success_probability` отображается в интерфейсе как «Вероятность успеха».
